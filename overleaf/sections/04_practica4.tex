\part{Práctica 4: Pruebas del Sistema (Testing)}

\section{Introducción}

El proceso de testing constituye una fase crítica dentro del ciclo de vida del software. Según Sommerville \cite{Sommerville2015}, las pruebas no solo buscan identificar defectos, sino demostrar que el sistema cumple con los requisitos especificados y que es adecuado para su propósito. En el contexto de la metodología UWE, el testing debe verificar tanto los aspectos funcionales (derivados de los casos de uso y diagramas de actividad) como los no funcionales (rendimiento, seguridad, usabilidad).

Esta práctica aborda la \textbf{auditoría integral} del sistema QR Restaurante, aplicando técnicas de prueba sobre cinco ejes fundamentales: usabilidad e interfaz, funcionalidad, base de datos, compatibilidad y rendimiento/seguridad. El objetivo es generar evidencias objetivas que demuestren la calidad del producto desarrollado.

\begin{enumerate}
    \item \textbf{Pruebas Unitarias}: Verificación individual de cada componente (validación de QR, cálculo de duración, recálculo de valoración media).
    \item \textbf{Pruebas de Integración}: Validación de la interacción entre módulos (escaneo → validación → registro de visita → actualización de aforo).
    \item \textbf{Pruebas de Sistema}: Evaluación completa del sistema desplegado contra todo el catálogo de requisitos.
    \item \textbf{Pruebas de Aceptación}: Verificación final desde la perspectiva de cada perfil de usuario.
\end{enumerate}

\subsection{Técnica de Diseño de Casos de Prueba}

Se emplea la técnica de \textbf{Partición de Equivalencia} combinada con \textbf{Análisis de Valor Límite} para maximizar la cobertura con el menor número de casos:

\begin{itemize}
    \item \textbf{Partición de equivalencia:} Se agrupan entradas similares que deben producir el mismo comportamiento (por ejemplo, todos los UUIDs inválidos se tratan igual).
    \item \textbf{Valor límite:} Se prueban los casos frontera (campo vacío, máximo de caracteres, límite de capacidad, 0 acompañantes, 11 acompañantes siendo el máximo 10).
\end{itemize}

\subsection{Metodología de Ejecución}

A continuación se describe el enfoque metodológico empleado para cada tipo de prueba:

\textbf{Pruebas de Base de Datos}: Se ejecutaron consultas SQL directamente contra la base de datos SQLite (\texttt{prisma/dev.db}) utilizando la herramienta \texttt{sqlite3}. Se verificó la integridad referencial mediante la inspección del esquema Prisma (constraints \texttt{ON DELETE CASCADE}, \texttt{FOREIGN KEY}, \texttt{UNIQUE}). El rendimiento de queries se evaluó con \texttt{EXPLAIN QUERY PLAN}.

\textbf{Pruebas de Seguridad}: Se realizó una auditoría estática de código (white-box testing) revisando todos los endpoints de la API. Se verificó: uso de consultas parametrizadas (Prisma), sanitización de entradas (Zod), control de acceso por roles (middleware JWT), exposición de datos sensibles (\texttt{select} explícito), y firma criptográfica de QR (HMAC-SHA256). No se requirió herramienta externa adicional.

\textbf{Pruebas Funcionales}: Se combinó revisión estática de código con ejecución manual de flujos. Se verificó cada requisito funcional (RF-01 a RF-04) contra su implementación en las rutas API correspondientes. Los flujos de aforo (RF-04) se validaron trazando la lógica de polling, cálculo de ocupación y componentes de UI.

\textbf{Pruebas de Rendimiento}: Se utilizó el build de producción de Next.js para medir \texttt{First Load JS} de cada página. Se analizaron las dependencias del proyecto y el tamaño del bundle. Las queries de base de datos se evaluaron con \texttt{EXPLAIN QUERY PLAN}.

\textbf{Pruebas de Compatibilidad}: Se verificó el uso de TailwindCSS con breakpoints responsive (\texttt{sm/md/lg}). Se confirmó la presencia de autoprefixer para compatibilidad cross-browser. Se inspeccionaron los metadatos de viewport y la accesibilidad ARIA mediante revisión de código.

\subsubsection{Detalle de hallazgos}

\begin{enumerate}
    \item \textbf{V-04 (Escaneo QR):} Severidad 2 — El visor de cámara no indica claramente qué hacer si el permiso de cámara es denegado. \textbf{Recomendación:} Mostrar un modal con instrucciones claras y botón para abrir configuración de permisos.

    \item \textbf{V-02 (Listado de Restaurantes):} Severidad 2 — Los filtros y la ordenación podrían confundir a usuarios noveles. \textbf{Recomendación:} Añadir texto de ayuda contextual (tooltip) junto a cada opción de filtro.

    \item \textbf{Todas las vistas:} Severidad 1 — Falta un indicador visual de estado de carga (skeleton screen o spinner) en las transiciones entre vistas. \textbf{Recomendación:} Implementar \textit{loading states} en cada punto de interacción asíncrona.
\end{enumerate}

FROM restaurantes r
WHERE r.tipo_cocina = 'italiana'
  AND r.capacidad_maxima > (
      SELECT COALESCE(SUM(v.acompanantes + 1), 0) 
      FROM visitas v 
      WHERE v.restaurante_id = r.id AND v.estado = 'activa'
  )
ORDER BY r.valoracion_media DESC
LIMIT 10;
-- Resultado esperado: tiempo < 500ms
\end{verbatim}

\subsection{Hallazgos de Base de Datos}

Se detectó la ausencia de índices sobre las columnas \texttt{visitas.estado} y \texttt{visitas.timestampEntrada}, las cuales son consultadas frecuentemente para el cálculo de aforo y generación de reportes. Con el volumen actual de datos (2 restaurantes, 6 visitas) no existe impacto en el rendimiento. Para un entorno de producción con más de 100.000 registros se recomienda crear los siguientes índices:

\begin{verbatim}
CREATE INDEX idx_visitas_estado ON visitas(estado);
CREATE INDEX idx_visitas_timestamp ON visitas(timestampEntrada);
\end{verbatim}

Todas las queries del proyecto utilizan \texttt{EXPLAIN QUERY PLAN} para verificar que los escaneos de tabla completa (\texttt{SCAN TABLE}) son inevitables solo cuando la cardinalidad es baja y el overhead de índice no se justifica.

\subsubsection{Protección del sistema de QR}

\begin{itemize}
    \item \textbf{Hash de validación}: Cada código QR incluye un hash HMAC-SHA256 generado con clave secreta del servidor. Al escanear, se verifica la integridad del contenido. La función \texttt{getSecret()} lanza un error si \texttt{QR\_HMAC\_SECRET} no está configurada, previniendo la falsificación de códigos QR (ver bug \#2 en \ref{sec:bugs}).
    \item \textbf{Expiración}: Los códigos QR pueden tener fecha de expiración configurable.
    \item \textbf{Unicidad}: Un UUID v4 garantiza que no se pueda adivinar otro código válido.
    \item \textbf{Rate limiting}: Se limitan los intentos de escaneo fallidos para prevenir fuerza bruta.
\end{itemize}

\section{Resultados y Conclusiones}

\subsection{Bugs Detectados y Corregidos}
\label{sec:bugs}

Durante la ejecución de las pruebas se identificaron y corrigieron los siguientes defectos:

\begin{enumerate}
    \item \textbf{capacidadMaxima=0 bloqueaba todas las entradas} (\texttt{visitas/route.ts:199}): Cuando un restaurante no definía límite de capacidad (\texttt{capacidadMaxima=0}), la condición de aforo completo se evaluaba incorrectamente, bloqueando todas las entradas. Se corrigió añadiendo la cláusula de guarda \texttt{capacidadMaxima > 0 \&\&} antes de la comparación.

    \item \textbf{getSecret() con fallback inseguro} (\texttt{qr-signing.ts:12}): La función de firma de QR retornaba un string vacío como valor por defecto cuando la variable de entorno \texttt{QR\_HMAC\_SECRET} no estaba configurada, lo que permitiría la falsificación de códigos QR. Se corrigió para lanzar un error explícito si el secreto no está definido.

    \item \textbf{Falta de metadatos viewport} (\texttt{layout.tsx}): El layout principal carecía de la meta etiqueta \texttt{viewport}, lo que impedía el escalado correcto en dispositivos móviles. Se añadió \texttt{export const viewport: Viewport = \{ width: 'device-width', initialScale: 1 \}}.
\end{enumerate}
