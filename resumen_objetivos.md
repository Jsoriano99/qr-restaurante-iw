# Contexto Global del Proyecto: Ingeniería Web (Guía para OpenCode)

Este documento resume los requisitos, fases y expectativas reales de la asignatura de Ingeniería Web, diseñado como contexto principal para el asistente de IA (OpenCode).

## 🎯 Objetivo Principal (El "Elefante en la Habitación")
Aunque la documentación oficial (Prácticas 1 a 4) se centra casi exclusivamente en la **Ingeniería del Software** (documentación, metodologías y modelado UML/UWE), el objetivo real y transversal de la asignatura es **desarrollar y programar una Aplicación Web completamente funcional** desde cero, trabajando en equipo mediante la metodología SCRUM.

**El asistente (OpenCode) DEBE saber que el usuario no solo necesita ayuda redactando memorias o diagramas, sino que necesita programar el Frontend, el Backend y la Base de Datos de la aplicación.**

---

## 🛠️ Fases del Proyecto (Cronograma de Prácticas)

El proyecto se divide en 4 grandes hitos evaluables, pero **la programación del código debe hacerse de forma continua e iterativa** durante todo el cuatrimestre.

### 1️⃣ Práctica 1: Contexto y Metodología (Gestión)
*   **Enfoque:** Configuración del entorno de trabajo y metodologías ágiles.
*   **Entregables:** Repositorio en GitHub, uso de SCRUM/SCRUMBAN, control de versiones (Git) y documentación en Markdown.
*   **Código:** El repositorio debe contener código funcional y documentado. Se definen roles en el equipo (Scrum Master, Desarrolladores Front/Back, QA, etc.).

### 2️⃣ Práctica 2: Especificación de Requisitos (Qué va a hacer la app)
*   **Enfoque:** Captura, definición y validación de requisitos usando el estándar UWE.
*   **Entregables:** 
    *   Catálogo de Requisitos: Información (RI), Funcionales (RF) y No Funcionales (RNF).
    *   **Diagramas UML:** Casos de Uso (funcionalidad general) y Actividades (flujos detallados).
*   **Acción para OpenCode:** Ayudar a traducir las ideas del usuario en requisitos formales y estructurar los diagramas.

### 3️⃣ Práctica 3: Modelado Arquitectónico en UWE (Cómo va a funcionar)
*   **Enfoque:** Diseño de la arquitectura de la aplicación usando MagicDraw/MagicUWE.
*   **Entregables (Modelos UWE):**
    *   **Contenido / Usuario:** Estructura de la información (Diagrama de Clases).
    *   **Navegación:** Nodos y enlaces (rutas de la aplicación).
    *   **Proceso:** Clases de procesos y diagramas de actividad.
    *   **Presentación:** Bocetos/wireframes abstractos de la interfaz gráfica.
*   **Acción para OpenCode:** A partir de aquí, OpenCode debe ayudar a **traducir estos modelos abstractos en código real** (ej. transformar el Modelo de Contenido en esquemas de Base de Datos, y el Modelo de Presentación en HTML/CSS/React/etc.).

### 4️⃣ Práctica 4: Testing y QA (Pruebas del Sistema)
*   **Enfoque:** Control de calidad de la **aplicación web finalizada**. No se programa funcionalidad nueva, se audita lo construido.
*   **Entregables (Memoria):** Evidencias y capturas de pruebas en 5 áreas: Funcionalidad, Bases de Datos, Compatibilidad, Rendimiento y Seguridad.
*   **Evaluación del Profesor:** Probará directamente la **Usabilidad** y la **Interfaz** de la web entregada.
*   **Defensa Oral:** Demostración en vivo de la aplicación funcionando.
*   **Acción para OpenCode:** Ayudar a corregir bugs detectados, optimizar el rendimiento, asegurar la base de datos y refinar la interfaz (UI/UX) para que sea sobresaliente.

---

## 🚀 Cómo debe actuar OpenCode a partir de ahora
1.  **Priorizar el Código:** Siempre que se definan requisitos (P2) o modelos (P3), OpenCode debe sugerir y guiar al usuario en la **implementación técnica** de los mismos en el repositorio.
2.  **Tecnologías:** Utilizar estándares modernos de desarrollo web, garantizando que el diseño sea atractivo, responsive y profesional (crucial para la P4).
3.  **Metodología:** Tratar cada tarea como un "Sprint", asegurando que cada funcionalidad diseñada teóricamente se programe y funcione en la práctica.
